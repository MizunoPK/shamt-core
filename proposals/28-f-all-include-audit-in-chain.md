## Problem

`/f-all` ‚Äî the framework-update meta-driver ‚Äî **deliberately excludes** `/f5-audit-framework` (Phase 6) from the chain it walks. The chain today is `validate(proposal) ‚Üí (optional /f2 + plan-validate) ‚Üí /f3 ‚Üí /f4 ‚Üí /f6`, and the exclusion is stated in four canonical surfaces: `host/templates/claude/commands/f-all.md` (three exclusion sites ‚Äî the chain ASCII block at "The chain" where Phase 6 is absent, the explicit rationale line ‚Äî *"Its sub-agent fan-out and f0-draft creation are surprising inside an unattended archive run; run it separately"* ‚Äî and the Exit-criteria bullet *"`/f5-audit-framework` was **not** run as part of the chain"*; the `/f5-audit-framework` mentions in its **Notes** section are Principle-1 / master-side-mirror pointers, **not** exclusion statements, and are left intact), its mirrored `host/templates/claude/skills/f-all/SKILL.md` (frontmatter description + chain summary + Exit-criteria line), `CLAUDE.md` ¬ß"How changes land" (the parenthetical `; /f5-audit-framework is excluded`), and `README.md` (the `/f-all` row in the Part-3 command table).

The operator wants the audit **back in the chain**: after a framework change is implemented (`/f3`) and regenerated (`/f4`) and before it is archived (`/f6`), `/f-all` should run `/f5-audit-framework` so the proposal's landing is automatically verified against the twelve audit dimensions rather than requiring a separate manual `/f5-audit-framework` invocation. The audit is already framed as *"post-implementation verification at Phase 6 of the framework-update flow"* (its own command description) ‚Äî so the slot exists; what's missing is `/f-all` actually dispatching it.

The original exclusion named two frictions, and this proposal **resolves** both rather than ignoring them: (1) the audit's only sub-agent spawn is the `audit-checker` clean-round confirmation (its sweep / auto-fix / f0-capture are inline primary work ‚Äî *not* fan-out), so it fits `/f-all`'s **existing driver-lifted-inner-persona topology** exactly as Phase 2 inline-validation already does, keeping exactly one nesting level; and (2) the audit's auto-fixes and f0 captures are **expected** in the normal Phase 5‚Üí6‚Üí7 flow ‚Äî they simply **fold into the `/f6` squash commit**, since running `/f5` before `/f6` is the standard framework-update sequence, not a surprise. See the Resolved Questions for the full reasoning.

**The audit body already anticipates this slot.** `host/templates/claude/commands/f5-audit-framework.md` already carries an "In-flow logging (folded into the proposal's landing commit)" mechanic (Step 3) stating that *"when the audit runs as Phase 6 of the framework-update flow ‚Ä¶ audit auto-fixes (and their regen output) and captured f0 stubs ride into the in-flight proposal's landing commit and squash-merge at `/f6-archive-proposal`"*. So this proposal does **not** invent the fold-into-`/f6` behavior ‚Äî it makes `/f-all` actually *dispatch* the audit into a slot the audit body already documents. The f5 body also contains **no `/f-all` reference and no exclusion statement**, so it needs **no edit** (it is not a fifth surface; the exclusion lives only in the four surfaces below).

## Risks

- **Regression / non-convergence risk** ‚Äî the audit's loop is the most complex phase; wired into the unattended chain it could fail to converge. Mitigated because `/f-all`'s existing strict-halt rule already covers a non-convergent inner-persona loop (a checker that never reaches `CONFIRMED`) ‚Äî the audit reuses that exact halt path, so a non-converging audit halts the run before `/f6` rather than spinning.
- **Scope-mixing (accepted by design)** ‚Äî the audit's auto-fixes (and any this-run f0 captures) fold into the proposal's `/f6` squash commit (OQ2). This is intentional: `/f5` before `/f6` is the normal flow, so the landed commit reflects the proposal *plus* the audit's clean-up of any drift that landing introduced. The land-commit message should still name the proposal as the primary change.
- **Topology risk (resolved)** ‚Äî the audit spawns only `audit-checker`, so it reuses `/f-all`'s existing driver-lifted-inner-persona dispatch (one nesting level); no new fan-out path is introduced. See OQ4.
- **Drift risk** ‚Äî the exclusion is stated in four canonical surfaces (f-all command ‚Äî multiple lines, f-all skill, CLAUDE.md, README); all must move together or the docs contradict each other (a D9 finding the audit itself would later flag).
- **Child-project compatibility** ‚Äî `/f-all` is master / self-host only, so child installs are unaffected beyond the routine `import-shamt` text refresh.

- ~~OQ1: Where does the audit run in the chain ‚Äî before or after `/f6`?~~ ‚Üí A: **Before `/f6`** (between `/f4` regen and `/f6` archive ‚Äî the literal Phase 5‚Üí6‚Üí7 order). Running `/f5` before `/f6` is the **normal framework-update flow**, not a special gate. The new chain is `validate ‚Üí /f3 ‚Üí /f4 ‚Üí /f5-audit ‚Üí /f6`.
- ~~OQ2: How are the audit's auto-fixes and f0-draft captures handled inside the unattended run?~~ ‚Üí A: **Folded into the proposal's squash commit** ‚Äî and this is **already the canonical behavior** the audit body specifies: `host/templates/claude/commands/f5-audit-framework.md` Step 3's "In-flow logging (folded into the proposal's landing commit)" already says that when the audit runs as Phase 6 of the framework-update flow, its auto-fixes (and regen output) and captured f0 stubs "ride into the in-flight proposal's landing commit and squash-merge at `/f6-archive-proposal`." So OQ2 is not a new design choice ‚Äî `/f-all` simply realizes the slot the audit body already documents. The audit runs its normal dual-track loop (auto-fix simple findings in place; capture intricate findings as f0 drafts); whatever it changes or creates rides into the single `/f6` squash commit alongside the proposal + regen output. No separate commit, no halt-on-finding ‚Äî the audit's own convergence (Pattern 1 clean exit) is what lets the chain advance to `/f6`. `/f-all` halts only if the audit cannot converge (its existing non-convergence halt).
- ~~OQ3: Always-on, or gated behind a `--audit` flag?~~ ‚Üí A: **Always-on.** Running `/f5` before `/f6` is the normal flow and use case ‚Äî no flag. Every `/f-all` run includes the audit phase.
- ~~OQ4: How does `/f-all` dispatch the audit without violating the one-nesting-level rule?~~ ‚Üí A: The audit spawns **exactly one** sub-agent ‚Äî the `audit-checker` on its clean round (verified against `host/templates/claude/commands/f5-audit-framework.md` Step 5); its sweep / auto-fix / f0-capture are all inline primary work, **not** fan-out (the old exclusion's "sub-agent fan-out" phrasing was imprecise). **On the f0-capture topology specifically:** f5 Step 3 phrases intricate-finding capture as "via `/f0-draft-proposal`", but `/f0-draft-proposal` is a pure stub-writer that **spawns no sub-agent** (verified against `host/templates/claude/commands/f0-draft-proposal.md` ‚Äî it has no Agent/dispatch). The audit primary captures an f0 draft by performing that **stub-write inline** ‚Äî it applies `/f0-draft-proposal`'s stub-writing *logic* directly (writes `proposals/{slug}.md` with the f0 status banner) ‚Äî **not** by dispatching a `/f0-draft-proposal` sub-agent. So f0-capture adds **no** nesting level; the `audit-checker` confirmation remains the audit phase's only sub-agent. The audit therefore slots into `/f-all`'s **existing driver-lifted-inner-persona topology**, identical in shape to Phase 2 inline-validation: the dispatched audit phase agent runs the audit's inline loop (Steps 1‚Äì4) to a clean round and **halts before Step 5** ‚Äî **without** invoking `/f5-audit-framework` (which would auto-spawn `audit-checker`, a forbidden 2nd nesting level); the **driver** then dispatches `audit-checker` itself; on any resettable finding the driver re-dispatches the primary to handle it (auto-fix / capture) and re-converge, then re-runs `audit-checker`, until `CONFIRMED`. **No persona edit** ‚Äî `audit-checker` is reused. **Resumability:** the audit leaves no disk artifact, but it is **idempotent** (a re-run on an already-clean framework converges immediately with no new fix / capture), so the Step-1 start discriminator `regen --check zero drift + branch exists` now routes to **Phase 6 (audit)** instead of Phase 7, and a resumed run **re-runs the entire harmless audit phase ‚Äî the primary's inline Steps 1‚Äì4 loop AND the driver-lifted `audit-checker` confirmation, not just the loop** (resume does not skip the checker; without a disk marker the driver re-derives Phase 6) before `/f6`. Both converge cleanly on the already-clean framework. The audit never prompts the user (it captures rather than halting), so it needs **no** open-question pause sentinel.
