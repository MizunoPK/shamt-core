## Problem

The framework-update flow is deliberately decomposed into one command per phase ‚Äî `/f1-propose-update` ‚Üí `/validate-artifact` ‚Üí (`/f2-plan-update-implementation`) ‚Üí `/f3-implement-update` ‚Üí `/f4-regen-framework` ‚Üí (`/f5-audit-framework`) ‚Üí `/f6-archive-proposal` ‚Äî per Principle 1 (`templates/SHAMT_RULES.template.md` ¬ß"Principle 1: Phase-per-command + slug resumability"). The "Phase N" numbers used throughout this proposal are the canonical ones the README ¬ß"Framework update flow (Part 3 ‚Äî master-side)" table assigns each command (Phase 0‚Äì7). That decomposition is correct for resumability, but it forces the operator to babysit a long, mostly-mechanical pipeline: after a proposal is drafted, the remaining phases are a sequence of `/clear` + per-phase command invocations the human runs one at a time ‚Äî threading the identifier through the chain (the slug for `/f2`/`/f3`/`/f6`, the proposal path for `/validate-artifact`, nothing for the slugless `/f4`) and watching each finish before starting the next. Most of those transitions carry no decision ‚Äî validation exited clean, so run implement; implement succeeded, so run regen ‚Äî yet each one costs a manual round-trip.

There is no driver that runs the remaining phases end-to-end. The directly applicable precedent is the **batch-validation handoff** (`README.md` line ~98; `reference/batch_validation_handoff.md`), which establishes the sanctioned pattern of a *stateless* orchestrator that fans out one sub-agent per artifact, each running the same Pattern 1 loop in its own context ‚Äî and, critically, solves the same nesting problem `/f-all` faces: its per-artifact primary sub-agent runs the phase's **inline instructions** rather than *invoking* the `/validate-artifact` command (which would auto-proceed to spawn its own checker, a forbidden second nesting level), and the **orchestrator lifts the checker-spawn up to itself** to keep "exactly one nesting level." `/f-all` is more involved than that handoff ‚Äî it sequences *distinct* phases rather than fanning out N copies of one, and for every step that internally needs its own sub-agent (any `/validate-artifact` run's adversarial checker ‚Äî the proposal validation, and the plan validation when `/f2` runs ‚Äî plus `/f3`'s `plan-executor`) it lifts that inner spawn up to the driver the same way (see Q5) ‚Äî but it inherits the same two properties that make the handoff compatible with the framework: **it holds no state of its own (every decision is re-derived from on-disk artifacts) and every underlying phase command remains independently runnable.**

This proposal adds `/f-all {slug}` ‚Äî a master-side driver that walks a proposal through every remaining framework-update phase, dispatching an **independent agent per phase** to carry out the real `/fX {slug}` work (invoking the command directly for self-contained phases; running the command's inline steps so the driver can lift the inner spawn up for phases that internally need a sub-agent ‚Äî see Q5), deriving the current phase entirely from on-disk artifacts (so it is itself slug-resumable), and **pausing only when a phase surfaces an open question for the user or halts on a failure it cannot resolve**. It is a convenience layer over the per-phase commands, not a replacement for them ‚Äî which is the line that keeps it compatible with Principle 1's "no single mega-orchestrator" / "no state file" clauses rather than contradicting them.

**Command-body requirements (row 1).** The `/f-all` command body must specify:

- **Master-only guard** ‚Äî child-side halt + redirect to the per-phase commands (mirrors `/f5-audit-framework`).
- **Phase resolution from on-disk artifacts** ‚Äî derive the current phase from artifact presence (proposal validation footer, plan presence, branch state, regen drift); no state file. This is what makes `/f-all` itself slug-resumable.
- **The chain** ‚Äî validate(proposal) ‚Üí (when the proposal carries the >10-ops note: `/f2` then validate the plan ‚Äî a single `/validate-artifact` run, or a batch-validation handoff when the plan is phase-decomposed) ‚Üí `/f3` ‚Üí `/f4` ‚Üí `/f6`. `/f5-audit-framework` is excluded (Q1).
- **Shared working tree** ‚Äî dispatch every sub-agent in the shared tree, never with worktree isolation; `/f3`'s `proposal/{NN}-{slug}` branch and canonical edits must be visible to `/f4` and `/f6`.
- **Dispatch topology (Q5, per `reference/batch_validation_handoff.md` ‚Äî exactly one nesting level)** ‚Äî self-contained phases (`/f2` authoring, `/f4`, `/f6`) are **invoked directly** in their own sub-agent. Phases that internally need a sub-agent (each `/validate-artifact` run's `validation-checker`; `/f3`'s `plan-executor`) run the command's **inline steps** and halt at the hand-off point; the **driver** then dispatches the inner persona itself. Naively dispatching a sub-agent to *invoke* `/validate-artifact` or `/f3` is the infeasible approach and must be ruled out in prose.
- **Pause vs. halt** ‚Äî there is exactly **one** resume-and-continue case: a phase surfacing a **structured open question** for the user (the driver surfaces it via `AskUserQuestion`, feeds the answer back, and re-dispatches ‚Äî Q2). *Every other* non-clean outcome ‚Äî failure, drift, **ambiguity the phase cannot resolve into a structured question**, non-convergence ‚Äî **halts** the flow and surfaces the report verbatim, with no retry (Q3). The distinction is the phase's own: a structured open-question ‚Üí pause; anything the phase reports as stuck/failed ‚Üí halt.
- **Pre-archive guard** ‚Äî before the autonomous `/f6` squash-merge, re-confirm `/f6`'s own pre-merge guards passed; never bypass them.

**Deliberately NOT touching `templates/SHAMT_RULES.template.md`:** `/f-all` is a master-side framework-update concept; it never appears in a child's installed rules, and the child rules file is size-budgeted (D12; cf. archived proposals 09 / 16). The Principle-1 reconciliation is therefore homed in **`CLAUDE.md`** (the master primer ‚Äî not size-budgeted, and the doc that already states the two principles are "inherited by every flow Shamt defines"), with a shorter pointer in `README.md`. Homing it in the master primer rather than the child rules is what keeps the canonical surface self-consistent: a future `/f5-audit-framework` D9 (contradiction) sweep that sees `/f-all` alongside Principle 1's "no single mega-orchestrator" finds the reconciliation in CLAUDE.md and does not flag it. (Revisit only if a contributor decides the Principle-1 *wording itself* must carry the carve-out ‚Äî deliberately avoided here to spare the child rules budget.)

## Risks

- **Regression risk** ‚Äî Low. Purely additive: a new command + skill + doc rows. No existing phase command changes behavior; each `/fX` remains independently runnable, so the manual path is untouched.
- **Drift risk** ‚Äî The new skill must mirror the new command body; `/f4-regen-framework` must run so `.claude/` picks up both new files. Standard new-managed-file drift, caught by `regenerate-framework.sh --check`.
- **Child-project compatibility** ‚Äî `/f-all` is master/self-host only (like `/f5-audit-framework`); in a child it halts with a redirect. Children pick up the new (inert-for-them) command on next `import-shamt` with no manual reconciliation.
- **Irreversible-action risk** ‚Äî `/f-all` ends at `/f6-archive-proposal` (Q1: full run), so the driver triggers a squash-merge + branch delete (an irreversible git mutation) **autonomously**, without a per-step human confirm. Accepted: the value is the unattended end-to-end run; mitigations are (a) the entire chain runs on the `proposal/{NN}-{slug}` branch so the merge target is recoverable via reflog, and (b) the driver halts before archive on any earlier-phase failure, so archive only runs on an otherwise-green pipeline. The command body must still re-confirm `/f6`'s own pre-merge guards passed (it does not bypass them).
- **Runaway / autonomy risk** ‚Äî A driver that dispatches agents across the chain could loop or burn tokens if a phase neither cleanly succeeds nor cleanly halts. Mitigated by the resolved strict-halt rule (Q3): the driver halts on any non-clean exit (not just open questions), never retries, and never re-dispatches a phase whose exit gate it cannot confirm from disk.
- **Shared-state / isolation pitfall** ‚Äî The chain depends on cross-phase git + disk state: `/f3` creates `proposal/{NN}-{slug}` and writes canonical edits that `/f4` (regen) and `/f6` (commit + squash-merge) must see. If the per-phase sub-agents were dispatched with **worktree isolation**, f3's branch/edits would be invisible to later phases and f6 would fail. The command body must mandate dispatch in the **shared working tree** (no `isolation: worktree`), and resumability deliberately relies on that shared on-disk state.
- **Nested-dispatch constraint (Q5)** ‚Äî A sub-agent cannot spawn another sub-agent, yet `/validate-artifact` (Standard) needs `validation-checker` and `/f3` may hand off to `plan-executor`. This is the **same** constraint `reference/batch_validation_handoff.md` already solves, and `/f-all` adopts its solution verbatim: the per-phase agent runs the phase's **inline instructions** (NOT a `/fX` command invocation, which would auto-proceed into the forbidden nested spawn) and halts at the hand-off point; the **driver** then dispatches the canonical inner persona (`validation-checker` / `plan-executor`) against the on-disk artifacts, keeping the topology at exactly one nesting level (driver ‚Üí phase agent; driver ‚Üí inner persona). This gives the driver a little loop-control responsibility for those steps (it participates in each validate adversarial-confirmation exit), which is why `/f-all` is *more* than the batch handoff ‚Äî but it stays stateless: same canonical personas, same on-disk artifacts, no new phase logic. **The command body must reproduce the batch-handoff topology explicitly** (inline-instruction phase agents + driver-lifted inner spawn); naively dispatching a sub-agent to *run the `/fX` command* is the infeasible approach and must be ruled out in prose. Getting this choreography right is the load-bearing build task for `/f3-implement-update`.
- **Open-questions debt** ‚Äî Boundaries, dispatch mechanism, halt semantics, and naming/scope are resolved in the Phase 1 dialog below before validation.

## Resolved Questions

- ~~Q1: Where does `/f-all` start and end; is f5-audit / f6-archive in scope?~~ ‚Üí A: **Validate ‚Üí Archive (full).** The chain runs Phase 2 (`/validate-artifact` on the proposal) ‚Üí Phase 3 (`/f2-plan-update-implementation`, only when the proposal carries the ">10 file ops" note ‚Äî **followed by validating the resulting plan** ‚Äî a single `/validate-artifact` run, or a batch-validation handoff when the plan is phase-decomposed) ‚Üí Phase 4 (`/f3-implement-update`) ‚Üí Phase 5 (`/f4-regen-framework`) ‚Üí Phase 7 (`/f6-archive-proposal`). **`/f5-audit-framework` is excluded** from the chain (optional verification; its sub-agent fan-out and f0-draft *creation* are surprising inside an unattended archive run ‚Äî run it separately when wanted). The driver **does** perform the irreversible squash-merge + branch delete autonomously, halting only for open questions / failures (accepted; see Risks).
- ~~Q2: Independent Agent sub-agents vs. a Workflow script?~~ ‚Üí A: **Sequential `Agent` dispatch.** The `/f-all` driver runs in the main agent loop and dispatches one independent `Agent` sub-agent per phase, in order; after each returns it inspects the result (clean ‚Üí dispatch next; open question ‚Üí surface via `AskUserQuestion`, then re-dispatch with the answer; failure ‚Üí halt). A `Workflow` was rejected because it runs to completion in the background and cannot pause mid-run for an open question ‚Äî the one behavior the request requires. (Note: this resolves the original "use workflows" framing toward sub-agents specifically; a Workflow is the wrong primitive when user pauses are mandatory.)
- ~~Q3: How aggressively may the driver remediate a non-clean phase?~~ ‚Üí A: **Strict halt, zero remediation.** Only a confirmable clean exit gate auto-advances (validation footer present; `/f3` reported success; `/f4` `--check` zero-drift; required artifact present). Any other outcome ‚Äî failure, drift, ambiguity, non-convergence ‚Äî halts immediately and surfaces the sub-agent's report verbatim. **The one exception is a structured open question** a phase surfaces for the user: that is the Q2 pause-and-resume case (surface via `AskUserQuestion`, feed the answer back, re-dispatch), *not* a halt. The boundary is the phase's own report ‚Äî a structured open question ‚Üí pause; anything the phase reports as stuck/failed/ambiguous ‚Üí halt. The driver never retries (not even "transient" failures) and never does design work; remediation past a halt is always the user's next move. This is the load-bearing runaway-safety rule.
- ~~Q4: Command name + scope?~~ ‚Üí A: **`/f-all`**, master/self-host-only. The hyphen-without-number marks it as a meta-driver spanning phases rather than a numbered phase. Scope is master-side only (the numbered / branch / squash-merge flow is inherently master-side); in a child it halts immediately and redirects to the per-phase commands ‚Äî mirroring `/f5-audit-framework`'s child-side behavior.
- ~~Q5: How does dispatch handle phases that internally need their own sub-agent (a sub-agent can't spawn another)?~~ ‚Üí A: **Driver orchestrates the inner step, via the established one-nesting-level topology of `reference/batch_validation_handoff.md`.** Each phase runs as its own sub-agent for fresh context. The mechanism (the part the first draft glossed): the per-phase agent is driven by the phase command's **inline instructions** and halts at the hand-off point ‚Äî it does **not** *invoke* `/fX` (invoking `/validate-artifact` would auto-proceed to Step 7 and spawn its own checker; invoking `/f3` would auto-hand-off to `plan-executor`) ‚Äî and the **driver** then dispatches the canonical inner persona (`validation-checker` / `plan-executor`) itself against the on-disk artifacts, exactly as the batch-validation orchestrator lifts the checker-spawn up to keep one nesting level. This is feasible precisely because the framework already ships and validates this pattern; "dispatch a sub-agent to run the `/fX` command and expect it to halt" is the infeasible reading and is explicitly ruled out. The same validate-style lift recurs for the conditional post-`/f2` plan validation, so the inner-spawn points are: proposal validation, plan validation (when `/f2` runs), and `/f3`'s builder (only on its architect/builder path ‚Äî `/f3`'s inline ‚â§10-ops path spawns nothing and is a direct-invoke phase). Consequence (folded into Problem + Risks): the driver carries a little loop-control responsibility for those steps, so `/f-all` is more than the batch handoff while remaining stateless and disk-derived.
