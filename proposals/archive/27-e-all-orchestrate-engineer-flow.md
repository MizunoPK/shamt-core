## Problem

The Engineer flow is deliberately decomposed into one command per phase — `/e1-start-story` → `/e2-define-spec` → (`/e3-plan-implementation` + `/e3b-write-testing-plan`) → `/e4-execute-plan` → `/e5-execute-tests` → `/e6-review-changes` → `/e7-resolve-feedback` → `/e8-finalize-story` — per Principle 1 (`templates/SHAMT_RULES.template.md` §"Principle 1: Phase-per-command + slug resumability"). The "Phase N" numbers used throughout this proposal are the canonical ones the README §"Engineer flow (Part 1 — story-level execution)" table assigns each command (Phase 1–8, plus the `/e3b` and `/e5b` sub-phases). That decomposition is correct for resumability, but — exactly as the framework-update flow's per-phase chain motivated `/f-all` (proposal `23-f-all-orchestrate-framework-update`, this branch) — it forces the operator to babysit a long pipeline: after a story is intaken, the remaining phases are a sequence of `/clear` + per-phase command invocations the human runs one at a time, threading the slug through the chain and watching each finish before starting the next. Many of those transitions carry no decision — the plan validated clean, so build; the build's verification passed, so test — yet each one costs a manual round-trip.

There is no driver that runs the story phases end-to-end. The directly applicable precedent is **`/f-all`**, the framework-update meta-driver (`host/templates/claude/commands/f-all.md`): a *stateless* orchestrator that runs in the main agent loop, dispatches one independent `Agent` sub-agent per phase in the shared working tree, derives the current phase entirely from on-disk artifacts (so it is itself slug-resumable), and pauses only when a phase surfaces a structured open question or halts on any other non-clean outcome. `/f-all` itself inherits the **batch-validation handoff** topology (`reference/batch_validation_handoff.md`): a phase that internally needs its own sub-agent runs the phase command's **inline instructions** rather than *invoking* the `/eX` command (which would auto-proceed to spawn its own sub-agent, a forbidden second nesting level), and the **driver lifts the inner spawn up to itself** to keep "exactly one nesting level."

This proposal adds `/e-all {slug}` — a driver that walks a single story through every remaining Engineer-flow phase, dispatching an **independent agent per phase** to carry out the real `/eX {slug}` work, deriving the current phase from on-disk artifacts, and **pausing only when a phase surfaces something the user must decide and halting on any failure it cannot resolve**. It is a convenience layer over the per-phase commands, not a replacement for them — the line that keeps it compatible with Principle 1's "no single mega-orchestrator / no state file" clauses. `/e-all` differs from `/f-all` in three structural ways that shape the design below: (1) it is **child-facing** — the Engineer flow is the story-execution flow every child runs, so unlike master-only `/f-all` there is no master-side guard; (2) it is **gate-heavy** — the Engineer flow has genuine interactive judgment gates (`/e2` Gate 2a design dialog and Gate 2b approval, `/e3` Gate 3 approval, `/e7` polish dialog, `/e8`'s explicit finalize confirmation) where `/f-all`'s chain mostly auto-advanced, so `/e-all` pauses far more often; and (3) it spans more inner personas — `validation-checker` (`/e2`, `/e3`, `/e3b`, `/e5b`, `/e6`), `plan-executor` (`/e4` build handoff — the Engineer flow's architect/builder split spans two phases, with the architect planning at `/e3` and the cheap-tier builder executing at `/e4`, so the `plan-executor` dispatch is `/e4`'s, unlike `/f-all`'s `/f3` which both plans and hands off within one phase), `user-simulator` and `test-executor` (`/e5`) — each lifted up to the driver under the same one-nesting-level rule.

**Command-body requirements (row 1).** The `/e-all` command body must specify:

- **Child-facing scope, no master guard** — unlike `/f-all`, `/e-all` runs wherever the Engineer flow runs (every child project, and master self-host). There is **no** Step-0 child halt. (This is the single biggest divergence from the `/f-all` template and must be called out so a contributor mirroring `/f-all` does not copy its master-only guard.)
- **Phase resolution from on-disk artifacts** — derive the **starting** phase once, at launch, from artifact presence (and, where no artifact exists, a disk/working-tree **gate**) under `stories/{slug}/` at **phase-boundary granularity** (the driver dispatches each phase as a sub-agent and blocks until it returns). Like `/f-all` — which resolves its `/f3` boundary via a working-tree diff-coverage gate and its `/f4` boundary via a `regen --check` zero-drift gate rather than a dedicated artifact — `/e-all` must use a gate for any phase that records no durable artifact. The discriminators, checked in order: `ticket.md` present → `/e1` done; `spec.md` footed (and `context.md` footed on Standard) → `/e2` done, with the spec's `Path:` header selecting Quick vs Standard for the rest of the chain; `implementation_plan.md` footed (Standard) **or** Quick-path-skipped → `/e3` done; `testing_plan.md` footed / inline checklist validated **or** TESTING_STANDARDS declares no suites → `/e3b` done/non-applicable; **`/e4` done is gate-derived, not artifact-derived** — `/e4-execute-plan` writes no "build complete" marker (its output is working-tree edits + commits per the plan's convention), so the discriminator is the **build gate**: the plan's `## Verification` section (or, on Quick path, the spec's Build Checklist) walks clean against the working tree, mirroring `/f-all`'s diff-coverage gate; `agent_test_session.md` verdict `Session PASS` (and `testing_plan.md` Results Log all PASS when suites exist) → `/e5` done; `feedback/review_vN.md` footed **or** (Quick-path no-issue) a `## Post-Build Review` block on the footed spec → `/e6` done; `feedback/addressed_feedback.md` dispositioned → `/e7` done; `ticket.md` carries `**Status: Done**` → `/e8` done (flow complete). No state file. A working tree matching none cleanly (a partially-applied phase left by an interrupted sub-agent — e.g. a build gate that would **not** walk clean) → **halt** rather than guess (strict-halt). Thereafter advance on each phase agent's returned report, not by re-deriving from disk.
- **The chain** — `/e1` → `/e2` → (`/e3` + `/e3b` + validate, **Standard path only**, and `/e3b` only when TESTING_STANDARDS declares suites) → `/e4` → `/e5` → `/e6` → `/e7` → `/e8`. Quick-path stories skip `/e3`/`/e3b` (build runs the spec's Build Checklist directly). `/e5b-write-manual-testing-plan` is **excluded** from the chain (optional, on-demand, orthogonal to the required Phase-5 agent-as-user run — mirrors `/f5`'s exclusion from `/f-all`).
- **Path + conditional detection** — the driver reads the spec's `Path:` header to know whether `/e3`/`/e3b` run, and reads `.shamt-core/project-specific-files/TESTING_STANDARDS.md` to know whether `/e3b` and the automated-suite half of `/e5` apply. These are mechanical disk/config reads, not user questions. If TESTING_STANDARDS.md declares suites but is itself missing/incomplete, the driver does not pre-flight-fail on it — the case surfaces at Phase 5, where the dispatched `user-simulator` returns its `Cannot run: TESTING_STANDARDS.md missing/incomplete — complete it first.` exit, which the driver classifies as a **halt** (the user completes/fixes the file and re-invokes).
- **Shared working tree** — dispatch every sub-agent in the shared tree, never with worktree isolation; the build's working-tree changes must be visible to `/e5` (test), `/e6` (review), `/e7` (polish), and `/e8` (commit).
- **Dispatch topology (per `reference/batch_validation_handoff.md` — exactly one nesting level)** — self-contained phases are **invoked directly** in their own sub-agent; phases that internally need a sub-agent (`/e2`/`/e3`/`/e3b`/`/e5b`/`/e6` validation → `validation-checker`; `/e4` build → `plan-executor`; `/e5` → `user-simulator` + `test-executor`) run the command's **inline steps** and halt at the hand-off point, and the **driver** then dispatches the inner persona itself. Naively dispatching a sub-agent to *invoke* `/eX` must be ruled out in prose.
- **Pause vs. halt, gate-heavy (Q-gates → headless + `AskUserQuestion`, exact `/f-all` mirror)** — every phase is dispatched as a sub-agent; the resume-and-continue cases are the Engineer flow's **interactive gates** plus any structured open question a phase surfaces, each lifted to the user via `AskUserQuestion`, the answer fed back, the phase re-dispatched. The driver branches on each dispatched persona's **already-shipped verbatim report contract** — `plan-executor`'s `All steps completed. Verification passed.` (advance) vs `Step [N] is ambiguous: …` (pause) vs `Step [N] failed: …` / `Plan defect at Step [N]: …` (halt); `test-executor`'s `All steps passed. Results logged.` (advance) vs `Step [N] is ambiguous: …` (pause — a structured open question lifted to the user, classified the same as `plan-executor`'s identically-named ambiguous exit) vs `Step [N] failed: Story bug — …` / `Test bug — …` / `Spec gap — …` / `Plan defect at Step [N]: …` / `Environment blocked at Step [N]: …` (halt); `user-simulator`'s `Session PASS` (advance) vs `Session BLOCKED: …` (halt) vs `Cannot run: TESTING_STANDARDS.md missing/incomplete — complete it first.` (halt — TESTING_STANDARDS declares suites but the file is missing/incomplete; the user completes/fixes it and re-invokes, never an autonomous fix, consistent with the strict-halt safety rule); `validation-checker`'s `CONFIRMED: Zero issues found after adversarial review.` (advance) vs any finding (re-dispatch primary, re-confirm). The **gates** — Gate 2a design dialog, Gate 2b / Gate 3 approvals, `/e7` polish dialog, `/e8` finalize confirmation — are the pause cases, surfaced via `AskUserQuestion` and re-dispatched on the answer. No persona edit is required (every contract above already ships). **Two row-1 authoring details the command body must spell out (both inherited from `/f-all`'s sentinel mechanics):** (a) the gate-bearing phases (`/e2`, `/e3`, `/e7`) run as *inline-instruction* sub-agents that **cannot** prompt the user themselves, so the driver's **dispatch prompt** must instruct each to **return** any gate prompt / unresolved Principle-2 open question to the driver using the verbatim `Open question: {q} | To resolve: {input needed}` sentinel `/f-all` defines (Gate 2a's 1–3 design options and the 2b/3/`/e8` approvals map cleanly onto one `AskUserQuestion` round-trip; multi-round design dialog is handled by successive re-dispatches off the now-updated on-disk artifact); (b) because each re-dispatch starts a fresh sub-agent context, the dispatch prompt must point the re-dispatched agent at the on-disk artifact (`spec.md` / `implementation_plan.md` / `feedback/review_vN.md`) so it resumes from saved state rather than lost conversation — the same disk-derived continuity the whole driver relies on.
- **Test-failure handling (Q-testloop → strict halt)** — the Engineer flow routes a Phase-5 failure to `/e7` for a root-cause loop, but `/e-all` does **not** drive that loop unattended. On any Phase-5 *failure* exit (`test-executor` `Step [N] failed: Story bug — …` / `Test bug — …` / `Spec gap — …` / `Plan defect at Step [N]: …` / `Environment blocked at Step [N]: …`; `user-simulator` `Session BLOCKED: …`) the driver **halts** and surfaces the executor's verbatim report; resuming via `/e7` (or fixing the build / plan) is the user's next move. (For `Environment blocked` specifically this is a **deliberate divergence** from the per-phase `/e5-execute-tests`, which does not halt but runs a one-question "which infrastructure piece is missing?" dialog and re-invokes `/e5` after the user resolves it externally; `/e-all` instead halts under the same strict-halt / no-autonomous-loop safety rule that keeps the `/e5`→`/e7`→re-`/e5` loop manual — the divergence is intentional, not an oversight of the per-phase dialog.) The one Phase-5 exit that is **not** a failure-halt is `test-executor`'s `Step [N] is ambiguous: …` — a structured open question lifted to the user as a **pause** (mirroring `plan-executor`'s identically-named ambiguous exit, and distinct from a test *failure*); the driver surfaces it via `AskUserQuestion` and re-dispatches Phase 5 against the on-disk testing artifact on the answer, exactly as the gate-pause cases do. This mirrors `/f-all`'s load-bearing runaway-safety rule — the driver never autonomously edits code to chase a green and never re-dispatches a phase whose exit gate it cannot confirm. (The flow's `/e5`→`/e7`→re-`/e5` loop stays a manual operation; the command body must state this explicitly so a contributor does not build an auto-fix loop.)
- **Terminal `/e8` guard** — before the (irreversible) commit + tracker-close, the driver must re-confirm `/e8`'s own three guards (prior phases complete; scoped clean tree; explicit user confirmation) are evaluated, not bypassed. `/e8`'s built-in explicit-confirm guard means the irreversible step is always user-gated — softer than `/f-all`'s autonomous squash-merge.

## Risks

- **Regression risk** — Low. Purely additive: a new command + skill + doc rows. No existing phase command changes behavior; each `/eX` remains independently runnable, so the manual path is untouched.
- **Drift risk** — The new skill must mirror the new command body; `/f4-regen-framework` must run so `.claude/` picks up both new files. Standard new-managed-file drift, caught by `regenerate-framework.sh --check`.
- **Child-project compatibility** — `/e-all` is child-facing (runs wherever the Engineer flow runs). Children pick up the new command + skill on next `import-shamt` with no manual reconciliation; it is immediately usable (no master-only inertness, unlike `/f-all`).
- **Irreversible-action risk** — `/e-all` ends at `/e8-finalize-story`, which commits the story and closes the tracker item (irreversible-ish). Unlike `/f-all`'s autonomous squash-merge, `/e8` carries its own **explicit user-confirmation guard**, so the irreversible step is always user-gated — the driver surfaces that confirm via `AskUserQuestion` (Q-gates) rather than performing it unattended, and re-confirms `/e8`'s other two guards (prior phases complete; scoped clean tree). `/e-all` spans the full e1→e8 chain (per the request — "all the stages for a single story"), so `/e8` is in scope, but its built-in confirm makes the irreversible step strictly safer than `/f-all`'s.
- **Runaway / autonomy risk** — A driver across e1→e8 could loop or burn tokens, especially around the Phase-5 test→`/e7`→re-test loop. Mitigated by the strict-halt rule and the resolved test-failure behavior (Q-testloop → halt): on any *failure* exit the driver halts, never retries silently, never autonomously edits code to chase a green, and never re-dispatches a phase whose exit gate it cannot confirm from disk. (The lone Phase-5 ambiguous exit is a user-gated pause, not an autonomous retry — the re-dispatch is driven by an `AskUserQuestion` answer, never by the driver itself.)
- **Gate-density risk** — the Engineer flow's many interactive gates mean `/e-all` pauses far more than `/f-all`. If gate handling were wrong, the "unattended" value would evaporate or the driver would make design decisions it should not. Resolved (Q-gates → headless dispatch + `AskUserQuestion` round-trips): each gate is a structured pause surfaced to the user and re-dispatched on the answer — the driver never makes a design call itself, it only sequences and relays.
- **Shared-state / isolation pitfall** — the chain depends on cross-phase working-tree state (the build's edits must reach test/review/polish/finalize). Dispatching any phase with worktree isolation would hide those edits. The command body must mandate the shared working tree.
- **Nested-dispatch constraint** — a sub-agent cannot spawn another sub-agent, yet several phases need inner personas. Same constraint `reference/batch_validation_handoff.md` solves; `/e-all` adopts the same one-nesting-level topology `/f-all` uses. Getting this choreography right (more personas than `/f-all`) is the load-bearing build task.
- **Open-questions debt** — gate handling, test-loop behavior, and the reconciliation home are resolved in the Phase 1 dialog below before validation.
