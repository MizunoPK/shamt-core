## Problem

`/f5-audit-framework` runs a dual-purpose sweep — it finds *simple* findings it can auto-fix in place and *intricate* findings that need a framework-update proposal. But the two tracks are not treated symmetrically. Simple findings are fixed and the loop continues; intricate findings **halt the loop**. The command body (`host/templates/claude/commands/f5-audit-framework.md` §Step 4) states: "Intricate findings routed to a proposal block the loop … State the routed proposal slug(s) and **halt the loop here**, directing the user to run the framework-update flow for them and re-audit afterward." The skill summary (`host/templates/claude/skills/f5-audit-framework/SKILL.md` bullet 13) and the reference (`reference/audit_dimensions.md` §"Worked examples — intricate", §"In-flow logging") carry the same framing: intricate ⇒ "route to `/f1-propose-update`, do not edit" and stop.

This means the very first proposal-worthy finding ends the audit, even when there are more simple fixes and more proposal candidates still undiscovered. A run that turns up one intricate finding and ten un-swept simple ones never reaches the simple ones. The user wants the audit to behave as a **single continuous improvement loop** that pursues *both* goals at once: keep applying small targeted completeness/correctness/consistency fixes **and** keep surfacing proposal candidates, halting only when it can find no more of *either*.

The blocker to "just keep going" is real, though: an intricate finding cannot be cleared by re-sweeping (its fix lives in an unimplemented proposal), so a naive "don't halt" loop would re-detect the same intricate finding forever and spin. The original design halted precisely to avoid this. The fix is to give the audit a way to **capture** an intricate finding as a durable artifact so a later sweep recognizes it as already-handled. That artifact is a lightweight **DRAFT proposal** — a new pre-proposal stage, **f0**: the audit quickly writes a proposal file marked as an unrefined draft (scratch notes + the implicated canonical files, much like the free-text blurb a user types into `/f1-propose-update`), then moves on **without** stopping to run the open-questions dialog. The dialog is deferred to whenever `/f1-propose-update {slug}` is later run on that draft.

For f0 drafts to be useful, `/f1-propose-update` must accept one as an input. Today f1 takes only a `{slug}` and seeds a blank proposal from `proposals/_template.md` (`host/templates/claude/commands/f1-propose-update.md` §Step 1); it has no notion of an intake blurb or an f0 draft to flesh out. This proposal formalizes two f1 input modes — a free-text blurb, and an existing f0 draft file — so the audit's f0 captures flow cleanly into a full f1 drafting pass.

**Scope — the audit (`/f5-audit-framework`) is master / self-host only.** The continuous loop and its auto-fix + f0-capture machinery run only where canonical sources are editable: the master / self-host target. A child project's `.shamt-core/` holds read-only imported copies of master, so the audit's two clearing actions are both unavailable there (it cannot auto-fix an imported copy, and there is nothing local to converge against). Under this proposal the only framework-update activities a child performs are **authoring proposals** — `/f0-draft-proposal` (quick capture) and `/f1-propose-update` (flesh out) — followed by `/sync-submit-proposal` to send them upstream, where implementation and the audit happen. Accordingly, the child-target *report-only audit mode* that exists today is **removed**: invoked in a child, `/f5-audit-framework` halts immediately and redirects the user to the f0 → f1 → sync-submit path. This both eliminates the child-side loop-termination problem (no child loop to spin) and matches the master/child contract that only proposals flow child → master.

## Risks

- **Regression risk — loop convergence.** The whole reason f5 halts today is that intricate findings can't be cleared by re-sweeping. The continuous loop keeps **Pattern 1's exit *shape*** (primary clean round + adversarial Haiku sub-agent) but adapts the *clean-round condition* for the capture track and converges on agent judgment: an intricate finding that already has an **addressing draft** in `proposals/` is "captured" and does **not** reset the loop or get re-drafted (the agent reads the proposals folder and judges this, exactly as it judges severity and finding-identity everywhere else in the loop). A round is clean when it needed no new auto-fix and no new draft — which, unlike canonical Pattern 1's "zero issues / one LOW" clean round, may still *report* captured intricate findings; this adaptation must be stated explicitly in the f5 body so the audit's own D2/D9 checks don't later flag it. The audit-checker is told which findings the primary captured this run so it likewise does not reset on an already-addressed one. This is **not** claimed to be a provably-cannot-spin mechanism — it is the same best-effort convergence Pattern 1 already relies on, no stronger and no weaker. The make-or-break property the validation loop must still hammer is that the convergence wording, the adapted clean-round, the capture/skip judgment, and the audit-checker carve-out are described consistently across the four self-describing audit files (see D9 risk). (Scope also narrows the surface: the loop runs **only** on master / self-host — there is no child loop, see Child-project compatibility.)
- **Duplicate-draft risk.** Across separate audit runs (and between the primary and the audit-checker within one run), the same intricate finding could be captured twice under different slugs if an agent fails to recognize an existing addressing draft. Mitigation: before capturing, the audit reads `proposals/` in all states and skips findings that already have an addressing draft (same judgment used to decide capture-vs-skip everywhere in the loop); the f0 slug-collision rule (append a numeric suffix, never overwrite — row 3) keeps any capture non-destructive. A rare duplicate draft is a harmless, user-reviewable outcome — not a loop-spin — since the user reviews drafts before running `/f1-propose-update` on them.
- **Drift risk (canonical vs generated `.claude/`).** Touches 7 host files (commands + skills + persona) plus the proposal template, the audit reference, and the cheatsheet. A missed regen leaves `.claude/` stale — itself a D1 finding. Rollback and Phase 5 must run regen.
- **D9 self-contradiction risk.** The audit's own behavior is described in four canonical places (f5 command, f5 skill, audit-checker persona, audit_dimensions reference). They must agree after the edit — including the *removal* of the child report-only mode (every one of the four must consistently say the audit is master / self-host only) — or the next audit flags a D9 contradiction in the audit's own definition. The f0 behavior is likewise described in two places (f0 command + f0 skill) and the _template marker convention — all must agree.
- **Child-project compatibility.** On next `import-shamt`, children pick up f0 + the reworked f5. The audit no longer runs in a child: invoked there, `/f5-audit-framework` halts and redirects to `/f0-draft-proposal` → `/f1-propose-update` → `/sync-submit-proposal`. This **removes** the prior child report-only audit mode — a deliberate behavior change matching the master/child contract (only proposals flow child → master). The one capability lost is audit-driven D1 drift detection in a child; it is covered instead by `/sync-import-shamt` (which re-runs regen on every pull) and by `/f4-regen-framework --check` for a manual local check. f0 *the command* still works in a child (drafting a new proposal file is not editing an imported canonical copy). No manual reconciliation expected.
- **Open-questions debt (resolved).** Seven genuine forks (f0 standalone vs inline; draft location/marker; convergence/dedup model; f0 pre-fill depth; child-target capture; whether the continuous loop runs on a child at all; what a child invocation of the audit does) were resolved in this proposal's open-questions dialog — see **Resolved Questions** — rather than left to the implementer. No design decisions remain open for Phase 4.

## Rollback Plan

1. `git revert <commit-sha>` on the canonical edit commit (this reverts the 8 EDITs and removes the 2 CREATEd f0 files from canonical sources).

4. Communication: note in the changelog that the audit reverts to halt-on-intricate.
